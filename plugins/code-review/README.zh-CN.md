# 代码审查插件

使用多个专用代理对拉取请求进行自动化代码审查，采用基于置信度的评分来过滤误报。

## 概述

代码审查插件通过启动多个并行代理从不同角度独立审查变更来自动化拉取请求审查。它使用置信评分来过滤误报，确保只发布高质量、可操作的反馈。

## 命令

### `/code-review`

使用多个专用代理对拉取请求执行自动化代码审查。

**功能：**
1. 检查是否需要审查（跳过已关闭、草稿、琐碎或已审查的 PR）
2. 从仓库收集相关 CLAUDE.md 指导文件
3. 总结拉取请求变更
4. 启动 4 个并行代理独立审查：
   - **代理 #1 和 #2**：审查 CLAUDE.md 合规性
   - **代理 #3**：扫描变更中的明显 bug
   - **代理 #4**：分析 git blame/历史以获取基于上下文的问题
5. 为每个问题评分 0-100 置信水平
6. 过滤低于 80 置信阈值的问题
7. 输出审查（默认输出到终端，或使用 `--comment` 标志作为 PR 评论）

**用法：**
```bash
/code-review [--comment]
```

**选项：**
- `--comment`：将审查作为评论发布到拉取请求上（默认：仅输出到终端）

**示例工作流程：**
```bash
# 在 PR 分支上，本地运行（输出到终端）：
/code-review

# 将审查发布为 PR 评论：
/code-review --comment

# Claude 将：
# - 并行启动 4 个审查代理
# - 为每个问题评分置信度
# - 输出 ≥80 置信度的问题（根据标志输出到终端或 PR）
# - 如果未发现高置信度问题则跳过
```

**功能：**
- 多个独立代理进行综合审查
- 基于置信度的评分减少误报（阈值：80）
- CLAUDE.md 合规检查，带有显式指导验证
- 专注于变更的 bug 检测（而非已有问题）
- 通过 git blame 进行历史上下文分析
- 自动跳过已关闭、草稿或已审查的 PR
- 使用完整 SHA 和行范围直接链接到代码

**审查评论格式：**
```markdown
## 代码审查

发现 3 个问题：

1. OAuth 回调缺少错误处理（CLAUDE.md 说 "始终处理 OAuth 错误")

https://github.com/owner/repo/blob/abc123.../src/auth.ts#L67-L72

2. 内存泄漏：OAuth 状态未清理（由于 finally 块中缺少清理导致的 bug）

https://github.com/owner/repo/blob/abc123.../src/auth.ts#L88-L95

3. 命名模式不一致（src/conventions/CLAUDE.md 说 "函数使用驼峰命名法")

https://github.com/owner/repo/blob/abc123.../src/utils.ts#L23-L28
```

**置信评分：**
- **0**：不自信，误报
- **25**：有些自信，可能是真的
- **50**：中等自信，真的但不重要
- **75**：高度自信，真的且重要
- **100**：绝对确定，肯定是真的

**过滤的误报：**
- PR 中未引入的已有问题
- 看起来像 bug 但不是的代码
- 挑剔的吹毛求疵
- linter 会发现的问题
- 一般质量问题（除非在 CLAUDE.md 中）
- 带有 lint ignore 注释的问题

## 安装

此插件包含在 Claude Code 仓库中。使用 Claude Code 时命令自动可用。

## 最佳实践

### 使用 `/code-review`
- 维护清晰的 CLAUDE.md 文件以获得更好的合规检查
- 信任 80+ 置信阈值 - 误报已被过滤
- 对所有非琐碎拉取请求运行
- 将代理发现作为人工审查的起点
- 根据重复的审查模式更新 CLAUDE.md

### 何时使用
- 所有有实质性变更的拉取请求
- 涉及关键代码路径的 PR
- 来自多个贡献者的 PR
- 指导合规性重要的 PR

### 何时不使用
- 已关闭或草稿 PR（自动跳过）
- 琐碎的自动化 PR（自动跳过）
- 需要立即合并的紧急热修复
- 已审查的 PR（自动跳过）

## 工作流程集成

### 标准 PR 审查工作流程：
```bash
# 创建带变更的 PR
# 本地运行审查（输出到终端）
/code-review

# 查看自动化反馈
# 进行必要的修复

# 可选：发布为 PR 评论
/code-review --comment

# 准备好后合并
```

### 作为 CI/CD 的一部分：
```bash
# 在 PR 创建或更新时触发
# 使用 --comment 标志发布审查评论
/code-review --comment
# 如果审查已存在则跳过
```

## 要求

- 具有 GitHub 集成的 Git 仓库
- 已安装并认证的 GitHub CLI (`gh`)
- CLAUDE.md 文件（可选但推荐用于指导检查）

## 故障排除

### 审查耗时太长

**问题**：代理在大 PR 上很慢

**解决方案**：
- 大变更的正常情况 - 代理并行运行
- 4 个独立代理确保彻底性
- 考虑将大 PR 分成较小的

### 误报太多

**问题**：审查标记不是真的问题

**解决方案**：
- 默认阈值是 80（已过滤大部分误报）
- 让 CLAUDE.md 更具体地说明重要内容
- 考虑标记的问题是否实际上有效

### 未发布审查评论

**问题**：`/code-review` 运行但未出现评论

**解决方案**：
检查是否：
- PR 已关闭（跳过审查）
- PR 是草稿（跳过审查）
- PR 是琐碎/自动化（跳过审查）
- PR 已有审查（跳过审查）
- 无问题评分 ≥80（不需要评论）

### 链接格式损坏

**问题**：代码链接在 GitHub 中渲染不正确

**解决方案**：
链接必须遵循此确切格式：
```
https://github.com/owner/repo/blob/[full-sha]/path/file.ext#L[start]-L[end]
```
- 必须使用完整 SHA（非缩写）
- 必须使用 `#L` 表示法
- 必须包含至少 1 行上下文的行范围

### GitHub CLI 不工作

**问题**：`gh` 命令失败

**解决方案**：
- 安装 GitHub CLI：`brew install gh`（macOS）或参阅 [GitHub CLI 安装](https://cli.github.com/)
- 认证：`gh auth login`
- 验证仓库有 GitHub 远程

## 提示

- **编写具体的 CLAUDE.md 文件**：清晰的指导 = 更好的审查
- **在 PR 中包含上下文**：帮助代理理解意图
- **使用置信评分**：≥80 的问题通常是正确的
- **迭代指导**：根据模式更新 CLAUDE.md
- **自动审查**：设置为 PR 工作流程的一部分
- **信任过滤**：阈值防止噪音

## 配置

### 调整置信阈值

默认阈值是 80。要调整，修改 `commands/code-review.md` 中的命令文件：
```markdown
过滤评分低于 80 的任何问题。
```

将 `80` 改为您偏好的阈值（0-100）。

### 自定义审查重点

编辑 `commands/code-review.md` 以添加或修改代理任务：
- 添加安全焦点代理
- 添加性能分析代理
- 添加可访问性检查代理
- 添加文档质量检查

## 技术细节

### 代理架构
- **2x CLAUDE.md 合规代理**：指导检查冗余
- **1x bug 检测器**：仅专注于变更中的明显 bug
- **1x 历史分析器**：来自 git blame 和历史的上下文
- **Nx 置信评分器**：每个问题一个用于独立评分

### 评分系统
- 每个问题独立评分 0-100
- 评分考虑证据强度和验证
- 阈值（默认 80）过滤低置信度问题
- 对于 CLAUDE.md 问题：验证指导明确提及它

### GitHub 集成
使用 `gh` CLI 用于：
- 查看 PR 详情和差异
- 获取仓库数据
- 读取 git blame 和历史
- 发布审查评论
