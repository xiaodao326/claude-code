# 功能开发插件

一个综合的结构化功能开发工作流程，配备用于代码库探索、架构设计和质量审查的专用代理。

## 概述

功能开发插件提供了构建新功能的系统性 7 阶段方法。它不是直接跳进代码，而是引导您理解代码库、提出澄清问题、设计架构并确保质量——从而产生更好设计的功能，与现有代码无缝集成。

## 理念

构建功能不仅仅是编写代码。您需要：
- **在变更前理解代码库**
- **提出问题**以澄清模糊的需求
- **深思熟虑后设计**再实现
- **构建后审查质量**

此插件将这些实践嵌入结构化工作流程，在使用 `/feature-dev` 命令时自动运行。

## 命令：`/feature-dev`

启动带有 7 个不同阶段的引导式功能开发工作流程。

**用法：**
```bash
/feature-dev 使用 OAuth 添加用户认证
```

或者简单地：
```bash
/feature-dev
```

命令将交互式地引导您完成整个过程。

## 7 阶段工作流程

### 阶段 1：发现

**目标**：理解需要构建什么

**发生什么：**
- 如果不清楚则澄清功能请求
- 询问您正在解决什么问题
- 识别约束和需求
- 总结理解并与您确认

**示例：**
```
您：/feature-dev 添加缓存
Claude：让我理解您需要什么...
        - 应该缓存什么？（API 响应、计算值等）
        - 您的性能需求是什么？
        - 您有偏好的缓存解决方案吗？
```

### 阶段 2：代码库探索

**目标**：理解相关现有代码和模式

**发生什么：**
- 并行启动 2-3 个 `code-explorer` 代理
- 每个代理探索不同方面（类似功能、架构、UI 模式）
- 代理返回带有要阅读的关键文件的综合分析
- Claude 阅读所有识别的文件以建立深入理解
- 展示发现的综合摘要

**启动的代理：**
- "查找与 [功能] 类似的功能并追踪实现"
- "映射 [区域] 的架构和抽象"
- "分析 [相关功能] 的当前实现"

**示例输出：**
```
找到类似功能：
- 用户认证（src/auth/）：使用 JWT 令牌、中间件模式
- 会话管理（src/session/）：Redis 支持的、24 小时过期
- API 安全（src/api/middleware/）：速率限制、CORS

需要理解的关键文件：
- src/auth/AuthService.ts:45 - 核心认证逻辑
- src/middleware/authMiddleware.ts:12 - 请求认证
- src/config/security.ts:8 - 安全配置
```

### 阶段 3：澄清问题

**目标**：填补空白并解决所有模糊之处

**发生什么：**
- 审查代码库发现和功能请求
- 识别未充分说明的方面：
  - 边缘情况
  - 错误处理
  - 集成点
  - 向后兼容性
  - 性能需求
- 以组织列表展示所有问题
- **等待您的回答后再继续**

**示例：**
```
在设计架构之前，我需要澄清：

1. OAuth 提供商：哪些 OAuth 提供商？（Google、GitHub、自定义？）
2. 用户数据：存储 OAuth 令牌还是仅用户档案？
3. 现有认证：替换当前认证还是添加在旁边？
4. 会话：与现有会话管理集成？
5. 错误处理：如何处理 OAuth 失败？
```

**关键**：此阶段确保设计开始前没有任何模糊之处。

### 阶段 4：架构设计

**目标**：设计多种实现方法

**发生什么：**
- 启动 2-3 个带有不同焦点的 `code-architect` 代理：
  - **最小变更**：最小改动、最大复用
  - **清洁架构**：可维护性、优雅抽象
  - **务实平衡**：速度 + 质量
- 审查所有方法
- 形成哪种最适合此任务的观点
- 展示比较带有权衡和建议
- **询问您偏好哪种方法**

**示例输出：**
```
我设计了 3 种方法：

方法 1：最小变更
- 使用 OAuth 方法扩展现有 AuthService
- 向现有认证路由器添加新 OAuth 路由
- 需要最小重构
优点：快速、低风险
缺点：将 OAuth 与现有认证耦合、更难测试

方法 2：清洁架构
- 新 OAuthService 带专用接口
- 分离 OAuth 路由器和中间件
- 重构 AuthService 使用公共接口
优点：清洁分离、可测试、可维护
缺点：更多文件、更多重构

方法 3：务实平衡
- 新 OAuthProvider 抽象
- 集成到现有 AuthService
- 最小重构、良好边界
优点：平衡复杂性和清洁度
缺点：仍有一些耦合

建议：方法 3 - 给您清洁边界而不需要
过度重构，并且很好地适合您现有架构。

您想使用哪种方法？
```

### 阶段 5：实现

**目标**：构建功能

**发生什么：**
- **开始前等待显式批准**
- 阅读前面阶段识别的所有相关文件
- 按选择的架构实现
- 严格遵循代码库约定
- 编写清洁、文档齐全的代码
- 进展时更新待办事项

**注意：**
- 实现仅在您批准后开始
- 遵循阶段 2 发现的模式
- 使用阶段 4 设计的架构
- 持续跟踪进度

### 阶段 6：质量审查

**目标**：确保代码简单、DRY、优雅且功能正确

**发生什么：**
- 并行启动 3 个带有不同焦点的 `code-reviewer` 代理：
  - **简单性/DRY/优雅性**：代码质量和可维护性
  - **Bug/正确性**：功能正确性和逻辑错误
  - **约定/抽象**：项目标准和模式
- 综合发现
- 识别最高严重性问题
- **展示发现并询问您想做什么**：
  - 现在修复
  - 稍后修复
  - 按现状继续
- 根据您的决定处理问题

**示例输出：**
```
代码审查结果：

高优先级问题：
1. OAuth 回调缺少错误处理（src/auth/oauth.ts:67）
2. 内存泄漏：OAuth 状态未清理（src/auth/oauth.ts:89）

中等优先级：
1. 可以简化令牌刷新逻辑（src/auth/oauth.ts:120）
2. 考虑提取 OAuth 配置验证

所有测试通过。代码遵循项目约定。

您想做什么？
```

### 阶段 7：摘要

**目标**：文档化已完成的内容

**发生什么：**
- 标记所有待办事项完成
- 总结：
  - 构建了什么
  - 做出的关键决策
  - 修改的文件
  - 建议的下一步

**示例：**
```
功能完成：OAuth 认证

构建内容：
- 支持 Google 和 GitHub 的 OAuth 提供商抽象
- 与现有认证集成的 OAuth 路由和中间件
- 令牌刷新和会话集成
- 所有 OAuth 流程的错误处理

关键决策：
- 使用带有 OAuthProvider 抽象的务实方法
- 与现有会话管理集成
- 添加 OAuth 状态以防止 CSRF

修改的文件：
- src/auth/OAuthProvider.ts（新）
- src/auth/AuthService.ts
- src/routes/auth.ts
- src/middleware/authMiddleware.ts

建议下一步：
- 为 OAuth 流程添加测试
- 添加更多 OAuth 提供商（Microsoft、Apple）
- 更新文档
```

## 代理

### `code-explorer`

**目的**：通过追踪执行路径深入分析现有代码库功能

**焦点领域：**
- 入口点和调用链
- 数据流和转换
- 架构层和模式
- 依赖和集成
- 实现细节

**触发时机：**
- 自动在阶段 2
- 探索代码时可手动调用

**输出：**
- 带有 file:line 参考的入口点
- 逐步执行流程
- 关键组件和职责
- 架构见解
- 要阅读的必要文件列表

### `code-architect`

**目的**：设计功能架构和实现蓝图

**焦点领域：**
- 代码库模式分析
- 架构决策
- 组件设计
- 实现路线图
- 数据流和构建顺序

**触发时机：**
- 自动在阶段 4
- 架构设计时可手动调用

**输出：**
- 发现的模式和约定
- 带有理由的架构决策
- 完整组件设计
- 带有特定文件的实现地图
- 带有阶段的构建顺序

### `code-reviewer`

**目的**：审查代码中的 bug、质量问题和项目约定

**焦点领域：**
- 项目指导合规（CLAUDE.md）
- Bug 检测
- 代码质量问题
- 基于置信度的过滤（仅报告高置信度问题 ≥80）

**触发时机：**
- 自动在阶段 6
- 编写代码后可手动调用

**输出：**
- 关键问题（置信度 75-100）
- 重要问题（置信度 50-74）
- 带有 file:line 参考的具体修复
- 项目指导参考

## 使用模式

### 完整工作流程（推荐用于新功能）：
```bash
/feature-dev 向 API 端点添加速率限制
```

让工作流程引导您完成所有 7 阶段。

### 手动代理调用：

**探索功能：**
```
"启动 code-explorer 追踪认证如何工作"
```

**设计架构：**
```
"启动 code-architect 设计缓存层"
```

**审查代码：**
```
"启动 code-reviewer 检查我最近的变更"
```

## 最佳实践

1. **对复杂功能使用完整工作流程**：7 阶段确保彻底规划
2. **深思熟虑地回答澄清问题**：阶段 3 防止未来困惑
3. **有意识地选择架构**：阶段 4 给您选择是有原因的
4. **不要跳过代码审查**：阶段 6 在问题到达生产前发现它们
5. **阅读建议的文件**：阶段 2 识别关键文件——阅读它们以理解上下文

## 何时使用此插件

**用于：**
- 涉及多个文件的新功能
- 需要架构决策的功能
- 与现有代码的复杂集成
- 求有些不清楚的功能

**不用于：**
- 单行 bug 修复
- 琐碎变更
- 定义明确、简单的任务
- 紧急热修复

## 要求

- Claude Code 已安装
- Git 仓库（用于代码审查）
- 带有现有代码库的项目（工作流程假设有现有代码可供学习）

## 故障排除

### 代理耗时太长

**问题**：代码探索或架构代理很慢

**解决方案**：
- 大代码库的正常情况
- 代理尽可能并行运行
- 彻底性以更好理解回报

### 太多澄清问题

**问题**：阶段 3 问太多问题

**解决方案**：
- 在初始功能请求中更具体
- 提前提供关于约束的上下文
- 如果真的没有偏好，说 "你认为最好的"

### 架构选项令人困惑

**问题**：阶段 4 中太多架构选项

**解决方案**：
- 信任建议——它基于代码库分析
- 如果仍不确定，请求更多解释
- 犹豫时选择务实选项

## 提示

- **在功能请求中具体**：更多细节 = 更少澄清问题
- **信任过程**：每个阶段建立在前一阶段上
- **审查代理输出**：代理提供关于您代码库的有价值见解
- **不要跳过阶段**：每个阶段有其目的
- **用于学习**：探索阶段教您关于您自己的代码库
